# Digging Skill Handoff

Digging은 복잡한 코드 변경, 리팩토링, 문서화, Firestore rules, 배포 전 검증처럼 위험도가 있는 작업을 Phase 단위로 천천히 진행하기 위한 실행 스킬이다. Split Plan이 지도를 만든다면, Digging은 그 지도를 따라 한 칸씩 실행하며 각 칸의 검증과 최종 회귀 QA를 빼먹지 않게 거는 안전 훅이다.

## 사용 시점

사용자가 다음과 같이 요청하면 Digging을 사용한다.

- "디깅으로 진행해"
- "땅굴 파듯이 천천히 해"
- "Phase별로 나눠서 끝까지 해"
- "검증 루프 돌리고 넘어가"
- "복잡한 리팩토링을 안정적으로 해"

## 핵심 절차

1. Context Dig
   - 최신 사용자 요청 확인
   - git status 확인
   - 관련 문서, 계획서, AGENTS.md, CLAUDE.md 확인
   - 관련 코드, 서비스, rules, 테스트, UI flow 확인

2. Split Plan 연동
   - Split Plan이 있으면 Digging은 Phase를 새로 설계하지 않고 기존 Phase를 실행 단위로 사용한다.
   - Phase 목표, Scope, 수정 대상, 성공 조건, 검증 방법, 위험/롤백 기준은 Split Plan을 기준으로 삼는다.
   - Split Plan이 없거나 너무 거칠면 작업 전 최소 Phase Map을 만든 뒤 사용자에게 공유한다.

3. Phase Execution
   - 한 번에 한 Phase만 진행한다.
   - 관련 없는 정리나 리팩토링은 하지 않는다.
   - 기존 사용자 변경은 되돌리지 않는다.
   - Phase 중 새 문제가 발견되면 현재 Phase 성공 조건에 직접 필요한 것만 수정하고, 나머지는 추가 이슈로 기록한다.

4. Phase Exit Hook
   - 각 Phase를 종료하거나 다음 Phase로 넘어가기 직전에 반드시 실행한다.
   - 이 훅을 통과하지 않으면 다음 Phase로 진행하지 않는다.
   - Phase 목표 대비 실제 수정 내용, 수정 파일, 계획과 달라진 점, 추가 발견 이슈를 짧게 기록한다.
   - Phase별 성공 조건과 검증 방법을 다시 확인한다.
   - Phase 검증 루프 2회를 실행하고 결과를 기록한다.

5. Two Validation Loops
   - 각 Phase마다 최소 2회 검증한다.
   - Loop 1: 직접 검증, 실패 수정, 재검증
   - Loop 2: 다른 각도에서 재검토, 개선, 재검증
   - 검증 중 수정 사항이 생기면 같은 검증을 다시 실행한다.
   - 검증을 실행하지 못했으면 그 이유와 대체 검증을 사용자에게 보고하고, 위험도가 크면 진행을 멈춘다.

6. Final Regression QA Hook
   - 전체 Phase가 끝난 뒤, 최종 완료 보고 전에 반드시 실행한다.
   - 웹/UI가 포함된 작업은 Playwright 기반 회귀 QA를 3회 루프로 수행한다.
   - 각 루프는 Playwright 검증, 문제 기록, 수정, 동일 검증 재실행 순서로 진행한다.
   - 3회 루프 중 수정 사항이 생기면 다음 루프는 수정 후 상태를 기준으로 다시 검증한다.
   - 마지막 루프에서 새 회귀 문제가 없어야 최종 완료로 보고한다.
   - 웹/UI 대상이 없어 Playwright가 맞지 않으면, 그 이유를 명시하고 lint, build, 테스트, rules dry-run 등 작업에 맞는 최종 회귀 검증을 3회 루프에 준해 수행한다.

## 강제 훅 체크리스트

Digging 중 아래 체크리스트를 스스로 확인한다. 하나라도 비어 있으면 완료 또는 다음 Phase 진행을 선언하지 않는다.

### Phase 종료 전

- 현재 Phase 목표와 성공 조건을 다시 읽었다.
- 실제 수정 내용과 수정 파일을 기록했다.
- 계획과 달라진 점 또는 추가 발견 이슈를 기록했다.
- Phase 검증 Loop 1을 실행했고, 실패가 있으면 수정 후 재검증했다.
- Phase 검증 Loop 2를 실행했고, 다른 각도에서 재검토했다.
- 다음 Phase로 넘어가는 근거를 확인했다.

### 최종 완료 전

- 모든 Phase의 종료 훅이 통과되었다.
- 최종 Playwright 회귀 QA 3회 루프를 실행했다. 웹/UI 대상이 없으면 대체 최종 회귀 검증 3회 루프를 실행했다.
- 각 QA 루프에서 발견한 문제, 수정 여부, 재검증 결과를 기록했다.
- 마지막 루프에서 새 회귀 문제가 발견되지 않았다.
- 남은 위험이나 미검증 항목이 있으면 최종 보고에 명시했다.

## 실행 로그 형식

각 Phase 종료 후 아래 형식으로 짧게 남긴다.

```text
Phase N 실행 로그
- 목표:
- 실제 수정:
- 수정 파일:
- 계획과 달라진 점:
- 추가 발견 이슈:
- 검증 Loop 1:
- 검증 Loop 2:
- 다음 Phase 진행 근거:
```

최종 완료 전 아래 형식으로 회귀 QA 결과를 남긴다.

```text
Final Regression QA
- Loop 1: 검증 / 발견 / 수정 / 재검증
- Loop 2: 검증 / 발견 / 수정 / 재검증
- Loop 3: 검증 / 발견 / 수정 / 재검증
- 최종 상태:
```

## 중단 조건

아래 상황은 사용자 승인 없이 진행하지 않는다.

- 실제 데이터 마이그레이션
- production Firestore rules 배포
- production Vercel 배포
- 기존 데이터를 삭제하거나 되돌리는 작업
- 제품 정책 판단이 필요한 변경
